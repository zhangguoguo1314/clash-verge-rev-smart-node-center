> 版本：1.0 | 日期：2026-07-02 | 基于：Clash Verge Rev v2.5.2 源码分析

## 1. 风险评估总览

| 风险等级 | 数量 | 说明 |
|---|---|---|
| 🔴 高风险 | 3 | 可能影响项目可行性或官方兼容性 |
| 🟡 中风险 | 5 | 需要特别注意和缓解措施 |
| 🟢 低风险 | 4 | 影响有限，常规处理即可 |

## 2. 高风险（🔴）

### 2.1 集成点的官方代码修改

| 风险项 | 详情 |
|---|---|
| 风险ID | R-HIGH-001 |
| 描述 | SNC 需要在 5 个官方文件中追加代码（lib.rs, _routers.tsx, main.tsx, _layout.tsx, verge.rs） |
| 影响 | 同步官方更新时可能产生 merge 冲突 |
| 概率 | 低（追加式修改冲突概率低，但官方重构这些文件时可能冲突） |
| 严重度 | 中（冲突解决通常简单，但需要手动处理） |

**缓解方案**：

| 方案 | 描述 | 优缺点 |
|---|---|---|
| 方案A（推荐） | 最小化集成，仅在 5 个位置追加代码 | 优点：改动最小，冲突概率低；缺点：仍需手动 merge |
| 方案B | 通过 Tauri 插件机制完全隔离 | 优点：零侵入；缺点：开发复杂度高，Tauri 插件限制多 |
| 方案C | 使用 Git patch/分支管理 | 优点：自动化处理冲突；缺点：增加 CI 复杂度 |

**建议**：采用方案A，同时在 CI/CD 中建立官方更新同步自动化流程，定期自动检测官方更新并运行 merge 测试。

---

### 2.2 Mihomo 内核 API 变更

| 风险项 | 详情 |
|---|---|
| 风险ID | R-HIGH-002 |
| 描述 | SNC 依赖 Mihomo 的 `select_node_for_group` 等 API，如果 Mihomo 更新改变 API 接口，SNC 可能失效 |
| 影响 | 自动切换、节点选择等核心功能失效 |
| 概率 | 低（Mihomo API 相对稳定，但不排除未来变更） |
| 严重度 | 高（影响 SNC 核心功能） |

**缓解方案**：

1. 通过 `tauri-plugin-mihomo` 插件与 Mihomo 通信，该插件由 Clash Verge Rev 维护，跟随官方更新
2. SNC 不直接调用 Mihomo REST API，而是通过 `tauri-plugin-mihomo` 的封装接口
3. 在 SNC 中增加 API 调用的错误处理和降级策略（API 不可用时通知用户而非崩溃）
4. 关键 API 调用增加版本检测，不兼容时给出明确提示

---

### 2.3 历史数据文件膨胀

| 风险项 | 详情 |
|---|---|
| 风险ID | R-HIGH-003 |
| 描述 | JSONL 文件存储历史数据，90 天 × 数百节点可能产生数百 MB 到数 GB 的数据 |
| 影响 | 磁盘空间占用、文件读取性能下降 |
| 概率 | 中（取决于节点数量和历史保留周期） |
| 严重度 | 中（影响用户体验，但不影响核心功能） |

**缓解方案**：

| 方案 | 描述 | 推荐度 |
|---|---|---|
| 方案A | 按天分文件 + 自动清理 + 数据压缩 | ⭐⭐⭐⭐ |
| 方案B | 引入 SQLite 嵌入式数据库 | ⭐⭐⭐（增加依赖，但性能好） |
| 方案C | 限制历史粒度（每小时聚合一次） | ⭐⭐⭐⭐（可与方案A结合） |
| 方案D | 限制最大文件大小 | ⭐⭐（治标不治本） |

**建议**：采用方案A + 方案C 组合。按天分文件存储，每小时聚合一次数据（而非每次测速都记录原始数据），超过保留周期的自动清理。

## 3. 中风险（🟡）

### 3.1 测速网络资源消耗

| 风险项 | 详情 |
|---|---|
| 风险ID | R-MID-001 |
| 描述 | 大量节点并发测速会消耗大量网络带宽和系统资源 |
| 影响 | 可能影响用户正常网络使用 |
| 概率 | 中 |
| 严重度 | 中 |

**缓解方案**：

1. 默认并发限制为 20 个节点
2. 下载测速使用小文件（1MB），可配置
3. 测速优先级低于用户正常流量（通过 Mihomo 规则控制）
4. 提供测速带宽限制选项
5. 允许用户配置测速时间（避免网络高峰期）

---

### 3.2 SNC 崩溃隔离

| 风险项 | 详情 |
|---|---|
| 风险ID | R-MID-002 |
| 描述 | SNC 后端 Rust 代码如果 panic，可能影响整个 Tauri 应用 |
| 影响 | 应用崩溃 |
| 概率 | 低（Rust 的错误处理机制，但 panic 仍可能发生） |
| 严重度 | 高 |

**缓解方案**：

1. 所有可能失败的异步操作使用 `tokio::spawn` + `catch_unwind` 隔离
2. SNC 核心管理器使用 `Arc<Mutex<Option<SncManager>>>` 包裹，崩溃时可重建
3. 关键路径增加错误处理，将 `unwrap()`/`expect()` 替换为适当的错误处理
4. 遵守官方 Clippy 规则（`unwrap_used = warn`）
5. SNC 定时任务使用独立的 tokio task，panic 不影响主线程

---

### 3.3 第三方依赖许可证冲突

| 风险项 | 详情 |
|---|---|
| 风险ID | R-MID-003 |
| 描述 | 新增第三方库可能与 GPL-3.0 许可证冲突 |
| 影响 | 法律风险 |
| 概率 | 低（主流 MIT/Apache 库通常兼容） |
| 严重度 | 高（许可证问题可能导致项目无法发布） |

**缓解方案**：

| 新增依赖 | 许可证 | GPL 兼容性 | 评估 |
|---|---|---|---|
| recharts | MIT | ✅ 兼容 | 安全 |
| SheetJS | Apache-2.0 | ✅ 兼容 | 安全 |
| rust_xlsxwriter | MIT | ✅ 兼容 | 安全 |
| calamine | MIT | ✅ 兼容 | 安全 |
| maxminddb | MIT | ✅ 兼容 | 安全 |
| uuid | Apache-2.0/MIT | ✅ 兼容 | 安全 |

**所有候选依赖均为 MIT 或 Apache-2.0 许可证，与 GPL-3.0 兼容。** 但每次引入新依赖前仍需确认许可证。

---

### 3.4 前端性能（大量节点渲染）

| 风险项 | 详情 |
|---|---|
| 风险ID | R-MID-004 |
| 描述 | Master Pool 可能有数百甚至上千节点，前端渲染性能可能受影响 |
| 影响 | 页面卡顿、高内存占用 |
| 概率 | 中 |
| 严重度 | 中 |

**缓解方案**：

1. 复用官方 `VirtualList` / `StickyVirtualList` 组件（@tanstack/react-virtual）
2. 节点列表分页加载（前端分页，每页 100 条）
3. 节点搜索和过滤在 Rust 后端执行（减少前端数据量）
4. Dashboard 只显示摘要数据，不渲染完整节点列表
5. 图表数据使用聚合后的数据（不渲染原始数据点）

---

### 3.5 功能开关失效

| 风险项 | 详情 |
|---|---|
| 风险ID | R-MID-005 |
| 描述 | SNC 关闭后，可能有残留后台任务或 UI 元素未完全清理 |
| 影响 | 与官方版本行为不完全一致 |
| 概率 | 低（通过良好的状态管理可避免） |
| 严重度 | 中（违反"关闭后与官方完全一致"的核心要求） |

**缓解方案**：

1. SNC Provider 在 `enabled=false` 时不渲染任何子组件
2. SNC 后端所有定时器在 `toggle_enabled(false)` 时立即停止
3. SNC 导航菜单在 `enabled=false` 时不显示
4. 增加集成测试：验证 SNC 关闭后，所有 SNC 相关行为停止
5. 增加 UI 快照测试：验证 SNC 关闭后的界面与官方一致

## 4. 低风险（🟢）

### 4.1 平台兼容性

| 风险项 | 详情 |
|---|---|
| 风险ID | R-LOW-001 |
| 描述 | 测速和检测功能在不同平台（Windows/macOS/Linux）的行为可能不一致 |
| 影响 | 跨平台功能差异 |
| 概率 | 低（reqwest 和 tokio 的跨平台一致性较好） |
| 严重度 | 低 |

**缓解方案**：SNC 使用的网络库（reqwest, tokio）具有良好跨平台支持。注意平台特定的 DNS 和网络配置差异。

---

### 4.2 国际化缺失

| 风险项 | 详情 |
|---|---|
| 风险ID | R-LOW-002 |
| 描述 | SNC 新增的 UI 文案可能未覆盖所有 13 种官方支持的语言 |
| 影响 | 部分语言显示英文 |
| 概率 | 中 |
| 严重度 | 低（可逐步完善） |

**缓解方案**：SNC 使用独立的 i18next 命名空间 `smart-node`，支持回退到英文，后续逐步翻译其他语言。

---

### 4.3 配置迁移

| 风险项 | 详情 |
|---|---|
| 风险ID | R-LOW-003 |
| 描述 | SNC 配置格式版本升级时，旧配置可能需要迁移 |
| 影响 | 旧配置无法使用 |
| 概率 | 低 |
| 严重度 | 低 |

**缓解方案**：配置文件包含 `version` 字段，启动时检测版本，不兼容时自动迁移或使用默认配置。

---

### 4.4 Excel 解析兼容性

| 风险项 | 详情 |
|---|---|
| 风险ID | R-LOW-004 |
| 描述 | 不同工具生成的 Excel 文件格式可能有差异 |
| 影响 | 部分文件无法正确解析 |
| 概率 | 中 |
| 严重度 | 低（非核心功能） |

**缓解方案**：使用成熟的 Excel 解析库（SheetJS / calamine），增加解析错误提示，支持 CSV/TXT 作为备选格式。

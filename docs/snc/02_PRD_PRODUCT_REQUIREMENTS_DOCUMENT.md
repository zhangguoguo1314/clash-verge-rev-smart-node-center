# Smart Node Center - 产品需求文档（PRD）

> 版本：1.0 | 日期：2026-07-02 | 基于：Clash Verge Rev v2.5.2 源码分析

---

## 1. 产品概述

### 1.1 产品定位

Smart Node Center（智能节点中心）是 Clash Verge Rev 的**插件式扩展模块**，为用户提供企业级的节点管理、智能分组、自动测速、自动切换和历史数据分析能力。

### 1.2 核心价值

| 价值维度 | 描述 |
|---|---|
| 智能化 | 自动分类、自动评分、自动切换、AI 推荐策略 |
| 可视化 | Dashboard 实时展示节点状态、测速历史、系统信息 |
| 高可靠 | 健康检测、自动修复、备用节点、Master Pool 兜底 |
| 高兼容 | 插件式设计，关闭后与官方版本完全一致 |
| 可维护 | 模块化架构，不影响官方功能，方便同步官方更新 |

### 1.3 设计原则

| 优先级 | 原则 | 说明 |
|---|---|---|
| P0 | 官方兼容性 | 不得修改官方已有源码、不得影响官方功能 |
| P0 | 稳定性 | 不得引入不稳定依赖，异常不得影响主程序 |
| P1 | 可维护性 | 代码结构清晰，方便后续同步官方更新 |
| P2 | 性能 | 测速、检测等异步执行，不阻塞主线程 |
| P3 | 扩展性 | 新功能可独立启停，便于后续扩展 |
| P4 | 功能数量 | 在满足以上前提下，尽可能完整实现需求 |

### 2.2 协议支持

#### 2.2.1 需求描述

Smart Node Center 需要识别和管理以下协议类型的节点。

#### 2.2.2 协议清单

| 协议 | 官方支持情况 | SNC 处理方式 |
|---|---|---|
| HTTP 代理 | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/http` |
| SOCKS 代理 | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/socks` |
| SS (Shadowsocks) | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/ss` |
| VMess | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/vmess` |
| VLESS | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/vless` |
| Trojan | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/trojan` |
| TUIC | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/tuic` |
| Hysteria2 | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/hysteria2` |
| WireGuard | ✅ Mihomo 原生支持 | 直接使用，URI 解析复用官方 `utils/uri-parser/wireguard` |

#### 2.2.3 复用说明

- **前端 URI 解析器**：`src/utils/uri-parser/` 已包含所有上述协议的解析器，**100% 复用**，无需新增代码
- **后端协议处理**：Mihomo 内核原生支持所有协议，SNC 只需将节点信息传递给 Mihomo
- **自动识别协议**：根据 URI scheme 自动识别（ss://, vmess://, vless://, trojan://, tuic://, hy2://, wg://, http://, socks5://）

#### 2.2.4 验收标准

- [ ] 支持解析所有 9 种协议的 URI
- [ ] 自动识别协议类型，无需用户手动选择
- [ ] 无法识别的协议给出明确提示，不静默忽略

| 格式 | 导入 | 导出 | 说明 |
|---|---|---|---|
| Base64 | ✅ | ✅ | 订阅链接常用编码格式 |
| YAML | ✅ | ✅ | Clash 配置文件格式（官方已有） |
| JSON | ✅ | ✅ | 结构化数据格式 |

#### 2.3.2 导入流程

```
用户选择文件/粘贴内容
    → 格式自动识别
    → 内容解析
    → 协议识别（复用 uri-parser）
    → 节点去重（与 Master Pool 比对）
    → 节点入库（写入 Master Pool）
    → 前端通知更新
```

#### 2.3.3 导出流程

```
用户选择导出范围（全部/选中/指定分组）
    → 用户选择格式
    → 数据序列化
    → 生成文件
    → 保存到指定位置
```

#### 2.3.4 新增依赖分析

| 依赖 | 用途 | 是否已有 | 许可证 | 说明 |
|---|---|---|---|---|
| Excel 导出 | .xlsx 读写 | ❌ 需新增（Rust 端） | 待确认 | Rust 端建议使用 `rust_xlsxwriter`（MIT）或 `calamine`（MIT）读取；CSV/TXT/JSON/YAML 均可用现有 serde 生态 |
| Excel 导出 | .xlsx 读写（备选） | ❌ 前端可处理 | - | 也可在前端使用 SheetJS（Apache-2.0）处理 Excel，避免增加 Rust 依赖 |

#### 2.3.5 验收标准

- [ ] 支持导入所有 6 种格式
- [ ] 支持导出所有 6 种格式
- [ ] 导入时自动去重，提示重复节点数量
- [ ] 导入时自动识别协议，无法识别的协议给出提示
- [ ] 导出时支持选择范围

#### 2.5.2 检测方式

| 方式 | 描述 | 实现方式 |
|---|---|---|
| HTTP | 发送 HTTP 请求，检查响应状态码 | reqwest GET 请求 |
| HTTPS | 发送 HTTPS 请求，检查响应状态码和证书 | reqwest GET 请求（TLS 验证） |
| TCP | 建立 TCP 连接，检查连通性 | tokio TcpStream 连接 |
| API | 调用特定 API 接口，验证返回内容 | reqwest GET 请求 + 响应内容匹配 |

#### 2.5.3 分组模型

```yaml
# smart_node_groups.yaml
groups:
  - id: "group_001"
    name: "Netflix 可用"
    description: "可访问 Netflix 的节点"
    detection:
      urls:
        - "https://www.netflix.com/title/80018499"
        - "https://www.netflix.com/"
      method: "https"
      timeout: 5000  # ms
      expected_status: 200
    auto_sync: true
    node_count: 15
    nodes:  # 引用 Master Pool 中的节点
      - "node_uid_001"
      - "node_uid_002"
    created_at: "2026-07-02T00:00:00Z"
    updated_at: "2026-07-02T00:00:00Z"
```

### 2.6 测速系统

#### 2.6.1 测速类型

| 类型 | 描述 | 优先级 |
|---|---|---|
| TCP 延迟 | TCP 握手延迟（ms） | P0 |
| HTTP 延迟 | HTTP 请求延迟（ms） | P0 |
| HTTPS 延迟 | HTTPS 请求延迟（ms） | P0 |
| TTFB | 首字节时间（Time To First Byte） | P1 |
| 下载速度 | 下载带宽测试（Mbps） | P1 |
| 上传速度 | 上传带宽测试（Mbps） | P2 |
| DNS 延迟 | DNS 解析延迟（ms） | P2 |
| Jitter | 延迟抖动（ms） | P2 |
| Loss | 丢包率（%） | P2 |
| 综合评分 | 加权综合评分 | P0 |

#### 2.6.2 测速调度

| 模式 | 描述 | 优先级 |
|---|---|---|
| 定时测速 | 按固定间隔自动测速 | P0 |
| 后台测速 | 不阻塞 UI 的后台测速 | P0 |
| 异步测速 | 多节点并行异步测速 | P0 |
| 测速缓存 | 缓存测速结果，避免重复测试 | P1 |
| 手动测速 | 用户手动触发单节点/批量测速 | P0 |

#### 2.6.3 综合评分算法（默认）

```
Score = W1 × LatencyScore + W2 × SpeedScore + W3 × StabilityScore + W4 × OnlineRateScore

其中：
- LatencyScore = f(latency)    # 越低越好，如 sigmoid 反函数
- SpeedScore = f(download)     # 越高越好
- StabilityScore = f(jitter, loss)  # 越稳定越好
- OnlineRateScore = f(online_rate)    # 越高越好
- W1=0.3, W2=0.25, W3=0.25, W4=0.2（默认权重，可配置）
```

#### 2.6.4 测速缓存策略

| 数据类型 | 缓存时间 | 说明 |
|---|---|---|
| TCP/HTTP 延迟 | 5 分钟 | 短期缓存，减少频繁测试 |
| 下载速度 | 30 分钟 | 速度变化较慢 |
| 综合评分 | 10 分钟 | 综合结果缓存 |
| 节点不可用状态 | 1 分钟 | 快速重试失败节点 |

#### 2.6.5 验收标准

- [ ] 支持 TCP、HTTP、HTTPS 延迟测试
- [ ] 支持下载速度测试
- [ ] 支持综合评分（权重可配置）
- [ ] 定时测速可配置间隔
- [ ] 后台测速不阻塞 UI
- [ ] 测速结果持久化

### 2.8 节点掉线处理

#### 2.8.1 掉线检测

| 规则 | 描述 |
|---|---|
| 失败 3 次 | 连续 3 次健康检测失败，判定为掉线 |
| 自动切换 | 掉线后自动切换到同分组下一个可用节点 |
| 备用节点 | 分组可配置备用节点，掉线时优先切换到备用 |
| 恢复检测 | 掉线节点定期尝试恢复检测 |
| 恢复切回 | 节点恢复后，根据配置决定是否自动切回 |
| Master Pool 兜底 | 分组全部失效时，从 Master Pool 选择可用节点 |

#### 2.8.2 掉线处理流程

```
健康检测失败
    → 计数器 +1
    → 计数器 >= 3?
        → 是：标记节点掉线
            → 同分组有可用节点?
                → 是：自动切换到下一个可用节点
                → 否：尝试备用节点
                    → 备用节点可用?
                        → 是：切换到备用
                        → 否：从 Master Pool 选择
            → 启动恢复检测（定期重试）
        → 否：等待下一次检测
```

#### 2.8.3 验收标准

- [ ] 连续失败 3 次自动切换
- [ ] 备用节点优先切换
- [ ] 全部失效时 Master Pool 兜底
- [ ] 恢复检测自动运行
- [ ] 恢复切回可配置

### 2.9 Master Pool

#### 2.9.1 设计要求

| 要求 | 描述 |
|---|---|
| 全局节点池 | Master Pool 保存所有已导入的节点 |
| 分组引用 | 所有分组通过节点 ID 引用 Master Pool 中的节点 |
| 不重复存储 | 节点只存储在 Master Pool 中，分组只存储引用 |
| 数据持久化 | Master Pool 数据持久化到文件，重启不丢失 |

#### 2.9.2 Master Pool 模型

```yaml
# smart_node_pool.yaml
pool:
  - uid: "node_001"
    name: "香港-01"
    protocol: "vless"
    address: "hk01.example.com"
    port: 443
    uri: "vless://xxx@hk01.example.com:443?..."
    source: "订阅A"
    source_url: "https://..."
    tags: ["香港", "AWS"]
    created_at: "2026-07-02T00:00:00Z"
    updated_at: "2026-07-02T00:00:00Z"
    last_health_check: "2026-07-02T00:05:00Z"
    status: "healthy"  # healthy / unhealthy / unknown
    score: 85
```

#### 2.9.3 验收标准

- [ ] Master Pool 存储所有节点
- [ ] 分组只存储节点引用（UID）
- [ ] 修改 Master Pool 节点信息后，所有分组同步更新
- [ ] 删除 Master Pool 节点时，所有分组自动移除引用

### 2.10 历史数据库

#### 2.10.1 存储内容

| 数据 | 保留周期 | 说明 |
|---|---|---|
| 24 小时 | 自动清理 | 实时数据 |
| 7 天 | 自动清理 | 短期历史 |
| 30 天 | 自动清理 | 中期历史 |
| 90 天 | 自动清理 | 长期历史 |

#### 2.10.2 统计指标

| 指标 | 描述 | 计算方式 |
|---|---|---|
| 成功率 | 节点检测成功次数 / 总检测次数 × 100% | 历史统计 |
| 失败率 | 节点检测失败次数 / 总检测次数 × 100% | 历史统计 |
| 测速记录 | 每次测速的延迟、速度等数据 | 原始记录 |
| 评分记录 | 每次评分的结果 | 原始记录 |
| 在线率 | 节点在线时间 / 总时间 × 100% | 基于健康检测记录 |
| 图表数据 | 自动生成时序图表 | 基于历史记录聚合 |

#### 2.10.3 数据存储方案

由于官方不使用数据库，SNC 也采用**文件存储**保持一致：

| 文件 | 格式 | 用途 |
|---|---|---|
| `smart_node_pool.yaml` | YAML | Master Pool 节点数据 |
| `smart_node_groups.yaml` | YAML | 分组配置 |
| `smart_node_config.yaml` | YAML | SNC 全局配置 |
| `smart_node_history/` | JSON Lines (JSONL) | 历史记录（按天分文件） |
| `smart_node_cache/` | JSON | 测速缓存 |

> **待确认**：历史数据量较大时（如 90 天 × 数百节点），JSONL 文件可能较大。是否需要引入轻量级嵌入式数据库（如 SQLite）？还是保持文件存储+定期清理策略？

#### 2.10.4 验收标准

- [ ] 历史数据按配置周期自动清理
- [ ] 统计指标准确计算
- [ ] 图表数据自动生成
- [ ] 数据存储不占用过多磁盘空间

## 3. 非功能需求

### 3.1 性能要求

| 指标 | 要求 |
|---|---|
| 测速并发数 | 最多 20 个节点同时测速 |
| 测速超时 | TCP/HTTP: 5s, 下载: 10s |
| Dashboard 刷新 | ≤ 2s |
| 历史查询 | ≤ 500ms（单节点 30 天数据） |
| 内存占用 | SNC 模块额外占用 ≤ 50MB |

### 3.2 可靠性要求

| 指标 | 要求 |
|---|---|
| SNC 崩溃不影响主程序 | SNC 异常被隔离，不影响 Clash Verge Rev 核心功能 |
| 配置文件损坏恢复 | 自动备份配置，损坏时从备份恢复 |
| 数据一致性 | Master Pool 与分组引用保持一致 |

### 3.3 兼容性要求

| 指标 | 要求 |
|---|---|
| 官方功能完整性 | 100% 不影响官方功能 |
| 配置文件兼容 | 新增配置独立文件，不修改官方配置文件格式 |
| 官方更新同步 | SNC 代码与官方代码物理隔离，方便 merge 官方更新 |
| 平台兼容 | 支持 Windows / macOS / Linux |
